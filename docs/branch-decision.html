TL;DR

Demo from feat/interview-prep. The question

With less than 24 hours to the Adobe interview, which branch should be the primary demo target?

  • origin/main — the pushed, submitted state. What Adobe has on their machine if they cloned the repo after submission. Contains the pipeline, the 8 ADRs, the Docker parity story, and the overlay fix pushed earlier today.
  • feat/interview-prep — local-only branch with four commits on top of main: the visible reuse story (Block B), the sourceType plumbing + RunSummaryPanel + D3 timing chart (Blocks C+D), the SPIKE-003 manual upload flow (local mode), and the interview narrative cheat-sheet.

Recommendation

Primary demo target: feat/interview-prep. The reasoning breaks down into five concrete arguments. None of them are "because it has more features" — that's not a good reason on its own. Each is tied to a specific interview failure mode on main.

  1. The reuse story is the single strongest demo beat in your whole assessment, and it is dark on main. The assignment text explicitly says "reuse input assets when available". On main, the reuse branch fires zero times on a fresh clone because nothing has an existingAsset pointing at a real file. You would be making the claim verbally with no visual proof. On feat/interview-prep, a reviewer watches the pipeline skip DALL-E for 3 of 6 creatives in real time and sees three independent visual signals: the emerald Reused badge on each card, the RunSummaryPanel's Reused: 3 / Generated: 3 count, and the PipelineTimingChart where reused creatives collapse to a compositing-only sliver. A reviewer cannot misinterpret this.
  2. The upload UI pre-empts an obvious objection. A reviewer skimming main will notice "the assignment says 'assets uploaded manually' but the form has no file picker". On feat/interview-prep the file picker is there, the two-step init + PUT flow works, magic-byte validation defeats Content-Type spoofing, and you have a SPIKE doc explaining the architectural decisions. If they ask "how would you wire manual upload?" — your answer is "here's the commit", not "here's how I would approach it."
  3. Everything on feat/interview-prep is strictly additive. No feature was removed, no API was broken, no existing test was deleted. The prompt builder walkthrough, the clean architecture story, the ADR-006 mappers demo — all identical. You gain visual and narrative surface without losing anything. There is no "risk" to the familiar parts of the story.
  4. The interview narrative cheat-sheet is organized around the feat/interview-prep feature set. docs/interview-narrative.md references the RunSummaryPanel, the D3 chart, the coastal-sun-protection campaign, the upload UI, and SPIKE-003. If you demo from main, roughly half the cheat-sheet does not apply and you would need to improvise the narrative at the worst possible time.
  5. "I kept working after submission" is a positive signal, not a negative one. The 48-hour assessment clock was the deadline for the submission. It was not a ban on continued development. A senior engineer who shipped on Sunday and continued iterating through Tuesday is demonstrating exactly the continuous-improvement mindset Adobe says they value. Do not apologize for this work. Frame it as deliberate post-MVP iteration.

Push to origin tonight as a backup (separate from the demo decision)

Right now feat/interview-prep is local only. If your laptop dies at 3am, the bootloader corrupts, someone else uses the machine, or you git reset --hard by accident, it's gone. That's 8+ hours of work at risk. 

git push -u origin feat/interview-prep

This does not affect main. It creates a new remote branch origin/feat/interview-prep that is visible in the GitHub branch list but does not touch what Adobe cloned. Your interview-prep work is now backed up to a second machine (GitHub's) for zero additional risk.

Framing to use on the call

Pick whichever flavor fits the conversation flow. The first one leads with the branch choice confidently; the second is what to say if a reviewer asks about it mid-call.

Opener — lead with it (confident, proactive)

"Quick orientation before I share my screen: I submitted the assessment on Sunday with the 48-hour deadline, and I've continued to iterate on the branch I'm about to show you. My submitted state on main has everything in the Adobe brief — the pipeline, the prompt builder, the 8 ADRs, the Docker parity story. My feat/interview-prep branch adds three things I wanted to get right post-submission: the reuse path is now visible end-to-end instead of architecturally-supported-but-invisible, I built the manual upload UI the assignment text calls for, and I wrote a SPIKE doc and an INVESTIGATION for the work. I'm going to demo from feat/interview-prep because the visible reuse story is the strongest beat, but if you'd prefer to see the submitted state verbatim I can switch to main anytime — they're the same underlying pipeline."

That's about 45 seconds of talking. Practice it once or twice. It frames the branch choice as deliberate engineering judgment, not as hiding something.

If they ask later — defensive but honest

"Everything you're seeing is on a feature branch I opened after submitting — I didn't want to ship with an incomplete reuse demo so I kept building. main on GitHub is the submitted state; this branch is post-submission polish. Happy to switch to main if that's more useful to you."

Two risks and their mitigations

Risk 1 — A strict reviewer says "I want to see exactly what you submitted"

Likelihood: low-to-moderate. Senior interviewers generally appreciate continued iteration; junior-focused or process-heavy reviewers may not.

Mitigation: have main ready in a second git worktree running on a different port before the call starts. If they ask, switch to the second browser tab — zero dead air. Do not git checkout branches mid-demo; it stops your dev server and creates awkward silence. Start both simultaneously: 

# Terminal 1 — feat/interview-prep on port 3000 (primary)
cd C:\dev\AI\Projects\Take-Home-Assessments\AdSpark
npm run dev

# Terminal 2 — main on port 3001 (fallback)
git worktree add ../AdSpark-main main
cd ../AdSpark-main
npm run dev -- --port 3001

Both are live. Two browser tabs. If the reviewer asks for main, flip tabs.

Risk 2 — Something built today regresses during the live demo

Likelihood: low. Gate is green, 282 tests pass, upload flow has been smoke-tested end-to-end in both local and S3 modes.

Mitigation: run Block F — a full live dry-run BEFORE the interview. Walk the exact click path from §1 of docs/interview-narrative.md at least once. If anything surprises you, catch it now, not tomorrow.

The bottom line

Action items tonight

1. Push feat/interview-prep to origin as a backup (git push -u origin feat/interview-prep). One command, zero cost, non-destructive. This is the single most important risk reduction step.

2. Set up a second git worktree with main running on port 3001 as a fallback in case a reviewer asks for the submitted state verbatim.

3. Read docs/interview-narrative.md end-to-end once (~15 minutes). Absorb the structure, memorize §0 opening beats and §10 "three numbers to remember cold."

4. Run the demo click path live at least once (Block F dry run). Use coastal-sun-protection as the primary sample brief.

5. Sleep. The hard work is already done. Your only job tomorrow is to explain what's already in front of you, honestly, without overselling or underselling.

Demo from feat/interview-prep. Keep main as a one-tab-away fallback. Open the call by framing the branch choice in your first 45 seconds. You have a strong story either way, but the visible reuse demo on feat/interview-prep is the single highest-leverage beat in your whole pitch — and you built it specifically to answer the question the assignment asked.
