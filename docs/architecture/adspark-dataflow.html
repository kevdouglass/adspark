1Entry point — the user clicks "Generate"

Two components can produce a brief: BriefForm (JSON editor) and BriefGeneratorAI (natural-language composer). Both converge on usePipelineState.submit() — the single hook that owns every state transition during a run.

BriefForm  ┐
       ├─>  usePipelineState.submit(body)  ─>  apiClient.generate(body)
BriefGeneratorAI ┘

State machine shape

// lib/hooks/usePipelineState.tsx
type PipelineState =
  | { status: "idle" }
  | { status: "submitting"; brief: GenerateRequestBody }
  | { status: "generating"; stage: PipelineStage }
  | { status: "complete";   result: GenerateResponseBody }
  | { status: "error";      error: ApiError };
Why discriminated unions? Because every UI component can ask state.status and TypeScript narrows the other fields for free — no null checks, no ?. chains, no "this shouldn't happen" bugs.

2Typed API client — the frontend's only way out

The UI never calls fetch() directly. It calls apiClient.generate(), which conforms to a GenerateFn interface. That interface is the seam where tests swap in a mock, Storybook swaps in a fixture, and production swaps in real HTTP. 

// lib/api/client.ts  (abridged)
export type GenerateFn = (
  body: GenerateRequestBody,
  signal?: AbortSignal
) => Promise<Result<GenerateResponseBody, ApiError>>;

export const apiClient: { generate: GenerateFn } = {
  async generate(body, signal) {
    const res = await fetch("/api/generate", {
      method: "POST",
      headers: { "content-type": "application/json" },
      body: JSON.stringify(body),
      signal,
    });
    return mapResponse(res); // converts HTTP → discriminated union
  },
};

3API route — thin, delegating, defensive

app/api/generate/route.ts is deliberately boring. It parses, validates, builds a RequestContext, calls runPipeline(), and maps the result. Roughly 50 LOC. Every byte of business logic lives in lib/. 

export async function POST(req: NextRequest) {
  const json = await req.json();

  // 1. Validate at the boundary (never trust the caller)
  const parsed = campaignBriefSchema.safeParse(json);
  if (!parsed.success) return toBadRequest(parsed.error);

  // 2. Build a per-request context (DI without a container)
  const ctx = buildRequestContext(req);

  // 3. Run the pipeline — the ONE call to the domain layer
  const result = await runPipeline(parsed.data, ctx);

  // 4. Map PipelineResult → HTTP wire format
  return toGenerateResponse(result);
}

4Zod validation at the boundary

campaignBriefSchema is the runtime source of truth for the brief shape. TypeScript types are derived from it via z.infer, so schema and types can never drift. 

// lib/pipeline/briefParser.ts
export const campaignBriefSchema = z.object({
  campaign: z.object({
    id:             z.string().min(1),
    name:           z.string().min(1),
    message:        z.string().min(1).max(140),
    targetRegion:   z.string(),
    targetAudience: z.string(),
    tone:           z.string(),
    season:         z.enum(VALID_SEASONS),
  }),
  products: z.array(productSchema).min(1),
  aspectRatios: z.array(aspectRatioSchema).min(1),
  outputFormats: outputFormatsSchema,
});

export type CampaignBrief = z.infer<typeof campaignBriefSchema>;
Why validate twice? The API route validates (so a bad payload becomes a clean 400), AND the pipeline re-validates at Stage 1. The second check is defensive — the pipeline should work from a unit test, a CLI, and a Next.js route with the same guarantees.

5RequestContext — dependency injection without a container

The pipeline is framework-agnostic. It doesn't know Next.js exists. Every external dependency (OpenAI SDK, storage provider, logger, timeouts) arrives through a typed bag called RequestContext, constructed once per request. 

// lib/api/services.ts
export interface RequestContext {
  openai:     OpenAI;
  storage:    StorageProvider;   // S3 or Local
  logger:     Logger;
  budgetMs:   number;            // staggered timeout
  requestId:  string;
  abortSignal: AbortSignal;
}
Why pass it instead of importing it? Tests construct a RequestContext with fakes and call runPipeline() directly — no Next.js mocking, no global state, no module-reset dance.

6Pipeline orchestrator — the timeline that never ships empty

lib/pipeline/pipeline.ts is the "main" of the pipeline. It composes six stages, threads RequestContext through each, and guarantees that a partial success is never discarded. 

export async function runPipeline(
  brief: CampaignBrief,
  ctx:   RequestContext,
): Promise<PipelineResult> {
  const t0 = Date.now();

  // Stage 1–2: parse + resolve existing assets
  const parsed   = parseBrief(brief);
  const resolved = await resolveAssets(parsed, ctx.storage);

  // Stage 3: build generation tasks (ONLY for products that need DALL-E)
  const tasks = buildGenerationTasks(parsed, resolved);

  // Stage 4: parallel DALL-E calls, partial failure tolerated
  const { successes, errors } = await generateImages(tasks, ctx);

  // Stage 5: text overlay (both reused + generated images)
  const composites = await overlayText(
    [...successes, ...reusedAsGenerated(resolved)],
    parsed.campaign.message,
  );

  // Stage 6: persist + build manifest
  const manifest = await organizeOutput(composites, ctx.storage);

  return {
    creatives: manifest.creatives,
    errors,
    totalTimeMs: Date.now() - t0,
  };
}
Partial failure guarantee: 5 successful images + 1 failed image returns 5 creatives + 1 error — never an empty result. The errors[] array is a first-class output field, not an exception.

7Stage 1 — briefParser (defensive re-check)

The pipeline re-validates the brief even though the API route already did. This catches calls from unit tests, CLIs, or anything else that bypasses the route. 

export function parseBrief(input: unknown): CampaignBrief {
  const result = campaignBriefSchema.safeParse(input);
  if (!result.success) {
    throw new BriefParseError(result.error);
  }
  return result.data;
}

8Stage 2 — assetResolver (the reuse branch)

The reuse branch is the cost-saver. If a product has existingAsset set and the storage layer can find it, we skip DALL-E entirely for that product and carry the buffer forward. 

export async function resolveAssets(
  brief: CampaignBrief,
  storage: StorageProvider,
): Promise<ResolvedProduct[]> {
  return Promise.all(brief.products.map(async (product) => {
    if (product.existingAsset) {
      const exists = await storage.exists(product.existingAsset);
      if (exists) {
        return {
          product,
          existingAssetBuffer: await storage.load(product.existingAsset),
        };
      }
    }
    return { product, existingAssetBuffer: null };
  }));
}
The badge comes from here. Every downstream stage sees whether existingAssetBuffer is null. The final sourceType field on the creative ("reused" vs "generated") is set based on that single check.

9Stage 3 — promptBuilder ★ (the star component)

Adobe Firefly cares about prompt engineering. So this component is deliberately template-based, not LLM-generated. A reviewer can read the template and predict exactly what DALL-E will receive — no "prompt of a prompt" black box. 

// lib/pipeline/promptBuilder.ts (abridged template)
const IMAGE_PROMPT_TEMPLATE = (v: TemplateVars) => `
A ${v.tone} product photograph of ${v.productName}, a ${v.category}.
${v.description}

Key features: ${v.keyFeatures.join(", ")}.
Target audience: ${v.targetAudience}.
Season/mood: ${v.season}.
Region: ${v.targetRegion}.

Composition: ${v.aspectHint}.
Lighting: ${v.lightingHint}.
Style: commercial advertising photography, shallow depth of field,
clean composition, hero product centered, photorealistic.

Do NOT include: text, logos, watermarks, people's faces, competitor brands.
`;

Why template-based?

  • Auditable: every variable is traceable to a brief field
  • Testable: unit tests assert the string contains expected phrases
  • Predictable: no LLM variance — same brief, same prompt, same time
  • Debuggable: bad outputs point at a specific template line

10Stage 4 — imageGenerator (parallel + retry)

DALL-E 3 calls are parallelized with p-limit. Concurrency is capped at DALLE_CONCURRENCY_LIMIT = 3 after tuning observed Tier 1 rate limiting at 5. 

const limit = pLimit(DALLE_CONCURRENCY_LIMIT);

const results = await Promise.allSettled(
  tasks.map((task) =>
    limit(() => withRetry(() => callDalle(task, ctx), {
      maxAttempts: 3,
      backoffMs: 500,
    }))
  )
);

// Partition into successes and typed errors — never throw
return partition(results);
Promise.allSettled, not Promise.all. The former isolates failures so one bad image doesn't cancel the others. If we used Promise.all, a moderation block on image #3 would lose images #4–6.

11Stage 5 — textOverlay (Sharp + Canvas)

Two libraries, each best-in-class at one job:

  • Sharp — crops DALL-E's square output to the requested aspect ratio (fastest resize in the Node ecosystem)
  • @napi-rs/canvas (Skia) — composites the campaign message with a gradient band + text shadow
const resized = await sharp(buffer)
  .resize({ width, height, fit: "cover" })
  .png()
  .toBuffer();

const canvas = createCanvas(width, height);
const ctx = canvas.getContext("2d");
const img = await loadImage(resized);

ctx.drawImage(img, 0, 0);
drawGradientBand(ctx, height);
drawCampaignMessage(ctx, message, { shadowBlur: 8 });

return canvas.toBuffer("image/png");
The Docker fontconfig bug (fixed): node:bookworm-slim ships zero fonts. Skia's fillText silently fell back to a zero-width glyph table — gradient drew, text was invisible. Fix: install fonts-dejavu-core + fontconfig + run fc-cache -f at image build time.

12Stage 6 — outputOrganizer + CreativeManifest

The manifest is the single source of truth for the dashboard. Every number shown in the UI is pre-aggregated server-side here — zero client math.

Class / Module Reference

Every top-level export, its responsibility, and a non-biased, hyper-critical review of what could have been done better. Ordered by the data flow.

Interview tip: if asked "what would you change?", pick one critique from below with a concrete remediation. Don't list all of them — that reads as unprioritized. Pick the one with the highest impact-to-effort ratio.
briefParser
lib/pipeline/briefParser.ts
Zod schema for CampaignBrief + a parseBrief(unknown) function that throws BriefParseError on failure.

Behaviors

  • Validates shape at the API boundary + inside the pipeline (defensive re-check)
  • Derives CampaignBrief type via z.infer so schema and types can't drift
  • Accepts unknown — usable from any upstream (route, CLI, test)

Hyper-critical critique

  • Schema + types + parser function live in one file — violates single-responsibility for a medium project
  • Zod default error messages are developer-friendly, not end-user-friendly
  • No JSON Schema export — external consumers can't generate typed clients

Could have been done better

  • Split into schema.ts + parser.ts + errors.ts
  • Add a formatZodError() layer that maps field paths to user-readable messages
  • Generate JSON Schema at build time for external API documentation
assetResolver
lib/pipeline/assetResolver.ts
resolveAssets(brief, storage) walks every product, checks for an existing asset, and returns a buffer (reuse) or null (generate).

Behaviors

  • Short-circuits DALL-E for products with a known asset
  • Returns ResolvedProduct[] — a union of reused + generate-me branches
  • Uses Promise.all across products for concurrent lookups

Hyper-critical critique

  • Exact filename match only. No fuzzy matching, no content-hash lookup, no slug-based fallback — fragile if a marketer misspells the asset name
  • storage.exists() + storage.load() is a sequential round-trip per product when it could be one operation that returns null on miss
  • No validation that the loaded asset is actually an image — a corrupt PNG silently passes through to textOverlay
  • No per-brief cache — running the same brief twice re-reads every file

Could have been done better

  • Add storage.tryLoad(key) that returns Buffer | null in one call
  • Validate via Sharp's metadata() before returning — fail fast on corruption
  • Accept an optional matchStrategy param (exact, slug, content-hash) to support fuzzy reuse
promptBuilder ★
lib/pipeline/promptBuilder.ts
The star component. Template-based DALL-E 3 prompt builder + aspect-ratio task emitter. The one thing Adobe Firefly will stress-test.

Behaviors

  • Injects brief variables into a deterministic template string
  • Emits one GenerationTask per (product, aspect-ratio)
  • Carries a static prohibited-terms list as the negative prompt
  • Aspect-ratio-specific composition hints (Feed / Story / Landscape)

Hyper-critical critique

  • Template is hardcoded in TypeScript. You can't A/B test, hot-reload, or version prompts without a deploy
  • No prompt versioning. The manifest records the final image but not which template rendered it — you can't bisect prompt regressions
  • No evaluation harness. I claim "auditable and testable" but there's no held-out brief set with quality scores to prove new templates improve anything
  • Prohibited terms are global. A luxury brand and a budget brand should have different negative prompts — currently they share one
  • No token budget check — a long description could push DALL-E's ~1000-char limit and silently truncate

Could have been done better

  • Extract templates to prompts/v1/product-hero.md with frontmatter + version
  • Record the template version on each Creative so the manifest is a prompt-change audit trail
  • Build a tiny eval harness: 20 reference briefs × 3 templates × human scores → regression detection
  • Accept a brand profile that supplies brand-specific positive + negative terms
imageGenerator
lib/pipeline/imageGenerator.ts
generateImages(tasks, ctx) fires parallel DALL-E 3 calls under a concurrency limit, with retry, returning successes + per-task errors.

Behaviors

  • Caps concurrency at DALLE_CONCURRENCY_LIMIT = 3 via p-limit
  • Retries each task up to 3 attempts with exponential backoff
  • Uses Promise.allSettled so one failure never cancels siblings
  • Partitions results into { successes, errors } — no throws

Hyper-critical critique

  • Retry doesn't distinguish error types. A 400 (bad prompt, unrecoverable) gets retried three times the same as a 429 (rate limit, transient). Wastes 2 attempts + time
  • No backoff jitter. If the whole batch hits a rate limit at the same time, they'll all retry at the same time and hit it again
  • Hardcoded concurrency. 3 was tuned by hand; an adaptive limit based on observed p95 latency would survive tier changes without a code change
  • No circuit breaker. 6 consecutive 503s still pays for 6 retries — a breaker would fail fast
  • Error metadata loses context — the failing task's prompt isn't attached to the error, so debugging requires correlating logs

Could have been done better

  • Classify errors: shouldRetry(err) returns false for 400/401/403, true for 429/5xx
  • Add jitter: baseBackoff * (1 + Math.random() * 0.3)
  • Wrap the client in a simple circuit breaker (opossum or a 30-line hand-roll)
  • Attach the originating task to errors so logs self-correlate
textOverlay
lib/pipeline/textOverlay.ts
overlayText(buffer, message) composites the campaign message on top of a Sharp-resized image via @napi-rs/canvas (Skia).

Behaviors

  • Sharp handles resize/crop to target aspect ratio
  • Canvas draws a gradient band + the message with a soft shadow
  • Word-wraps long messages and scales font to fit
  • Returns a composite PNG buffer ready to save

Hyper-critical critique

  • Font discovery is OS-dependent. The Docker-fontconfig bug is a symptom — a real production version would embed fonts directly via registerFont(), not rely on Skia's fontconfig lookup
  • Word-wrap is naive. Breaks on whitespace only — no RTL support, no CJK breakpoints, no grapheme cluster awareness
  • Gradient band position is fixed. A product photo with dark subject matter at the bottom makes the white text illegible — the overlay should sample image luminance and pick a better location or flip to dark text
  • No brand font injection — every creative uses DejaVu regardless of brand identity

Could have been done better

  • Register fonts at boot via registerFont('./fonts/Inter.ttf') — eliminates the OS dependency
  • Use Intl.Segmenter for grapheme-aware word-wrap
  • Sample a 20×20 region for luminance, pick text color + position adaptively
  • Accept a brand object with { fontUrl, textColor, bandPlacement }
outputOrganizer
lib/pipeline/outputOrganizer.ts
Writes each composite to output/{product}/{ratio}/ and builds the typed CreativeManifest.

Behaviors

  • Delegates file writes to the injected StorageProvider
  • Aggregates per-creative timings into a summary block
  • Flags partial-success runs in the manifest status

Hyper-critical critique

  • Manifest is in-memory only. It's returned via HTTP but never written to storage — you can't re-hydrate a past run without re-running the pipeline
  • No content-addressable naming. Re-running the same brief overwrites creatives silently instead of producing a new version — history is lost
  • No signed URL generation for S3 outputs — frontend gets raw S3 URLs that only work if the bucket is public

Could have been done better

  • Write manifest.json next to creatives — one file describes the whole run
  • Use output/{brief-hash}/{timestamp}/ for immutable history
  • Storage provider exposes signedUrl(key, ttl) for private buckets
runPipeline orchestrator ★
lib/pipeline/pipeline.ts
Composes the six stages, threads RequestContext through, and guarantees partial-success semantics.

Behaviors

  • Enforces the stage execution order
  • Routes reused vs generated branches to text overlay uniformly
  • Aggregates errors without throwing — returns a structured result

Hyper-critical critique

  • Stage sequence is hardcoded. A new stage (e.g. moderation pass) means editing this file — not a DAG where stages declare dependencies
  • Timeout budget is passed down but not enforced downstream. No stage checks "do I still have time?" before starting expensive work — they just inherit the top-level signal
  • No observability hooks. No span boundaries, no structured trace events — "what took so long?" needs manual log grep
  • Error aggregation loses stage attribution — an ImageProcessingError in the errors array doesn't cleanly say which task it belongs to

Could have been done better

  • Represent stages as { name, run, dependsOn } objects and run them through a topological executor
  • Each stage checks ctx.remainingMs() before starting and bails with a budget-exceeded error
  • Wrap each stage in an OTel span so "time to reused" vs "time to generated" becomes a first-class metric
RequestContext
lib/api/services.ts
Per-request dependency bag: OpenAI client, storage provider, logger, timeout budget, request id. Constructed once in the API route.

Behaviors

  • Hand-wires all dependencies the pipeline needs
  • Makes the pipeline testable via fake injection

Hyper-critical critique

  • Not a proper DI container. It's a struct — no scopes, no lazy resolution, no lifecycle management
  • No async disposal. The storage provider might hold handles; nothing calls dispose() on request end
  • Hand-wired in the route. Every new dependency means editing the construction site — easy to forget a field

Could have been done better

  • Move construction to lib/api/services.ts with named factory per environment
  • Add an AsyncDisposable hook and call it in a route-level try/finally
  • Or bite the bullet on a mini-DI library (tsyringe, awilix) if the wiring grows past ~5 deps
timeouts
lib/api/timeouts.ts
Four staggered constants — per-image → pipeline → client → Vercel cap.

Behaviors

  • Each layer's timeout is shorter than the layer above, so typed errors surface before opaque ones

Hyper-critical critique

  • Static constants. Tuned for Tier 1 — a Pro deploy with a 300s cap should use different values, but the only way to do that today is edit and redeploy
  • Values aren't coupled to observed latency. If DALL-E p95 drifts, the cascade becomes either too tight (false timeouts) or too loose (runs eat the whole budget)

Could have been done better

  • Read budgets from env vars with sensible defaults
  • Expose a /metrics endpoint reporting p50/p95 so the constants can be data-driven
ApiError / error mapper
lib/api/errors.ts
Discriminated-union ApiError with a stable cause field. Frontend branches on discriminant, not message text.

Behaviors

  • Classifies pipeline errors into safe-to-return shapes
  • Separates user-facing messages from internal diagnostic metadata

Hyper-critical critique

  • Cause list grows by hand. Every new error type means adding a discriminant in two places — the class and the mapper — easy to drift
  • No error code stability guarantee. Renaming a discriminant is a silent breaking change for frontend consumers

Could have been done better

  • Use a Zod enum for cause — single source of truth
  • Add a CHANGELOG for error codes treated as a versioned API surface
usePipelineState
lib/hooks/usePipelineState.tsx
MVI state hook. Discriminated-union state, reducer-driven transitions, single source of truth for every dashboard component.

Behaviors

  • Manages idle → submitting → generating → complete/error transitions
  • Exposes a submit(body) action that calls the API client

Hyper-critical critique

  • Reducer is inline with the hook. Grows harder to test as states multiply — a separate module with a pure reducer is more unit-friendly
  • Error state is a dead end. No retry affordance — the user has to reload or re-trigger the whole form
  • No optimistic rollback. If submit fails mid-flight, we transition to error but we've already cleared the previous results

Could have been done better

  • Extract pipelineReducer to its own file + add pure-function tests
  • Add a retry() action that re-sends the last body
  • Keep the previous complete state around as lastResult so error doesn't wipe it
apiClient
lib/api/client.ts
Typed fetch() wrapper conforming to GenerateFn. The only network code on the frontend.

Behaviors

  • POSTs to /api/generate with a 58s timeout
  • Maps HTTP responses to a Result<Ok, Err> union
  • Honors AbortSignal for cancellation

Hyper-critical critique

  • Fixed timeout. Same budget for a 1-product brief as a 6-product brief — should scale to input size
  • No client-side retries. Transient network failures (flaky wifi) force the user to re-click manually
  • No request/response timing telemetry. The frontend can't tell how long the API took vs overhead

Could have been done better

  • Pass a timeout multiplier derived from body.products.length * body.aspectRatios.length
  • Add a thin retry layer for network errors only (not HTTP 4xx)
  • Wrap fetch with a performance.now() pair and expose timings via an onTiming hook
StorageProvider (Local + S3)
lib/storage/*.ts
Interface with two implementations. Factory picks one based on STORAGE_MODE. Read-only seed-dir fallback supports the reuse branch in Docker.

Behaviors

  • exists(), load(), save() — minimal surface
  • LocalStorage enforces path-traversal guards + symlink escape checks

Hyper-critical critique

  • Interface is too thin. No list(), no stat(), no stream() — callers can't introspect or handle large files
  • S3Storage has no retry on upload. A flaky network during the write step fails the entire run
  • No TTL / cleanup. Local output/ grows forever; nothing evicts old runs

Could have been done better

  • Expand interface with list(prefix) and stream() — S3 already supports both cheaply
  • Wrap S3 SDK calls in the same retry helper used for DALL-E
  • Add a sweep() helper and a cron to age out runs > N days
multi-agent orchestration
lib/ai/agents.ts
Four-phase stakeholder review: triage → draft → 4 parallel reviewers → synthesis. Powers BriefGeneratorAI.

Behaviors

  • Separate LLM call per persona so voices don't average out
  • Promise.all fan-out during the review phase
  • Synthesis step explains why each reviewer was accepted

Hyper-critical critique

  • Reviewer list is hardcoded. You can't add a stakeholder without editing this file — should be a config-driven pipeline
  • No streaming. The user waits ~8–12s with a spinner; streaming each phase as it completes would feel dramatically faster
  • No cost tracking. Six LLM calls per click with no telemetry — easy to silently overspend
  • Synthesis is the authoritative step, but if it fails we silently fall back to the draft. The user never learns the reviewers had input

Could have been done better

  • Load personas from config/reviewers.json
  • Stream via Server-Sent Events or React Server Components so each phase lands as it completes
  • Emit a cost_cents metric per phase for visibility
  • On synthesis failure, surface the reviewer notes anyway so the user can merge manually
RunSummary + dashboard components
lib/pipeline/runSummary.ts · components/*
Server-side aggregation of the manifest into counts + status. Read by RunSummaryPanel, CreativeGallery, and PipelineTimingChart.

Behaviors

  • Pre-aggregates reused/generated/failed counts so UI does zero math
  • Produces the status discriminant (complete / partial / failed)

Hyper-critical critique

  • Summary is coupled to the UI. Fields exist because the dashboard wants them (reusedAssets) — a proper audit log would carry more than the UI needs
  • No time-series view. One run = one snapshot — you can't compare two consecutive runs of the same brief
  • CreativeGallery has no virtualization. Fine at 6 tiles, janky at 60
  • BriefForm is 400+ lines. Should be decomposed into subforms (campaign, products, output)

Could have been done better

  • Split "audit summary" (complete) from "UI summary" (aggregated subset)
  • Persist runs in SQLite/Postgres for history and comparisons
  • Use react-window in the gallery once tile count grows
  • Decompose BriefForm into CampaignSection + ProductsSection + OutputSection

2. State management — discriminated union ↔ sealed interface

Both platforms converge on the same pattern: an immutable state object represented as a closed set of variants. The difference is only syntax.

Web — TypeScript + React hook
// lib/hooks/usePipelineState.tsx
type PipelineState =
  | { status: "idle" }
  | { status: "submitting" }
  | { status: "generating";
      stage: PipelineStage }
  | { status: "complete";
      result: GenerateResponseBody }
  | { status: "error";
      error: ApiError };

export function usePipelineState() {
  const [state, dispatch] = useReducer(
    pipelineReducer, { status: "idle" }
  );

  const submit = async (body) => {
    dispatch({ type: "SUBMIT", body });
    const r = await apiClient.generate(body);
    dispatch({ type: "RESULT", result: r });
  };

  return { state, submit };
}
Android — Kotlin + ViewModel + StateFlow
// feature/brief/BriefViewModel.kt
sealed interface PipelineState {
  object Idle : PipelineState
  object Submitting : PipelineState
  data class Generating(
    val stage: PipelineStage
  ) : PipelineState
  data class Complete(
    val result: GenerateResponse
  ) : PipelineState
  data class Error(
    val error: ApiError
  ) : PipelineState
}

@HiltViewModel
class BriefViewModel @Inject constructor(
  private val generate: GenerateCreativesUseCase,
) : ViewModel() {

  private val _state = MutableStateFlow<PipelineState>(Idle)
  val state: StateFlow<PipelineState> = _state.asStateFlow()

  fun submit(brief: CampaignBrief) = viewModelScope.launch {
    _state.value = Submitting
    _state.value = when (val r = generate(brief)) {
      is Result.Ok  -> Complete(r.value)
      is Result.Err -> Error(r.error)
    }
  }
}
Exhaustiveness parity: TypeScript narrows a discriminated union by state.status. Kotlin narrows a sealed interface by when (state) is Idle -> .... Both compilers yell if you miss a branch. That's the same guarantee, just spelled differently.

4. Domain layer — pipeline is framework-agnostic, so it ports almost 1:1

This is the payoff for keeping the pipeline pure. briefParser, assetResolver, promptBuilder, and the orchestrator have no React, no Next.js, no fetch — they're just functions over data. Translating them to Kotlin is a syntax rewrite, not a redesign.

Web — TypeScript
// lib/pipeline/assetResolver.ts
export async function resolveAssets(
  brief: CampaignBrief,
  storage: StorageProvider,
): Promise<ResolvedProduct[]> {
  return Promise.all(
    brief.products.map(async (product) => {
      if (!product.existingAsset)
        return { product, buffer: null };
      if (!(await storage.exists(product.existingAsset)))
        return { product, buffer: null };
      return {
        product,
        buffer: await storage.load(product.existingAsset),
      };
    })
  );
}
Android — Kotlin (:core:domain)
// core/domain/pipeline/ResolveAssetsUseCase.kt
class ResolveAssetsUseCase @Inject constructor(
  private val storage: StorageProvider,
) {
  suspend operator fun invoke(
    brief: CampaignBrief,
  ): List<ResolvedProduct> = coroutineScope {
    brief.products.map { product ->
      async {
        val key = product.existingAsset
          ?: return@async ResolvedProduct(product, null)
        if (!storage.exists(key))
          return@async ResolvedProduct(product, null)
        ResolvedProduct(product, storage.load(key))
      }
    }.awaitAll()
  }
}
Promise.all → coroutineScope + async + awaitAll. Same semantics (fan out, wait for all, fail fast on exception), different vocabulary. The pipeline orchestrator in Kotlin would chain these with suspend functions under a viewModelScope or a dedicated CoroutineScope if executed outside a ViewModel.

3. Anticipated Q&A

Architecture & tech stack very likely

Why Next.js? Why not a Python CLI like OP's spec implied?
Three reasons. One, the JD explicitly called out React/Next.js — meeting the team in their stack beats writing in mine. Two, a Next.js app ships a visible product in the same codebase as the pipeline — reviewer opens Vercel, sees it working. Three, a CLI is harder to demo than a UI. I documented this as ADR-001.
Why direct OpenAI SDK and not MCP?
ADR-002. MCP would add a deploy target, latency, and a failure mode without adding any evaluation signal for this assessment. The simplest thing that works is a direct SDK call. If this were production and I had multiple clients integrating, MCP would be on the table — but not for a 48-hour spike.
Walk me through your timeout strategy.
Four staggered constants in lib/api/timeouts.ts: DALL-E per-image 45s → pipeline budget 50s → client fetch 58s → Vercel hard cap 60s. Each layer fails before the layer above it, so you always get a typed error naming the stage that ran out, not an opaque 504. Known limitation: DALL-E 3 Tier 1 p90 latency can exceed the per-image budget under load — I documented that in the README and the route file.
How do you handle partial failure?
imageGenerator uses Promise.allSettled, not Promise.all. If 5 of 6 images succeed and 1 fails, the response returns 5 creatives and a per-task error in the errors[] field — never an empty result. The RunSummaryPanel renders a partial-success pill when this happens.

The star — prompt engineering very likely

Walk me through your prompt builder. Why templates instead of LLM-generated prompts?
Four reasons: auditable — every variable traces back to a brief field; testable — unit tests assert the string contains expected phrases; predictable — same brief, same prompt, zero LLM variance; debuggable — a bad output points at a specific template line. An LLM-built "prompt of a prompt" gives you none of those. For Firefly specifically, a reviewer needs to be able to read and reason about the prompt — templates are the only shape that supports that.
How did you verify the prompts actually work?
__tests__/promptBuilder.test.ts asserts: structure (all sections present), variable injection (brief fields appear in output), prohibited-terms enforcement (no text/logos/faces in the negative list), and aspect-ratio-specific composition hints. ADR-008 captures the broader strategy — testing shape and invariants, not exact strings, because LLM outputs are non-deterministic even at temperature=0.
How would you swap DALL-E for Firefly?
The pipeline is already decoupled from the SDK through the imageGenerator stage — it accepts a function that returns GeneratedImage. Swapping providers is a single file change, plus updating prompts to Firefly's idiosyncrasies (it responds differently to some directives). In production I'd A/B them — Firefly for commercial-safety reasons, DALL-E for fallback.

Reuse branch + cost very likely

The assignment says “reuse input assets when available.” Where does that live?
lib/pipeline/assetResolver.ts, Stage 2. If a product has existingAsset set and storage can find it, we skip DALL-E entirely and carry the buffer forward. The sourceType field on each creative ("reused" vs "generated") is set here. In the UI, it shows up twice: the green pill on each tile (visual) and the Reused: N count in the RunSummaryPanel (audit). Dashboard's timing chart makes the savings visible as a shape — reused bars collapse to the compositing sliver because generationTimeMs = 0.
Where does the “reused” decision actually get made?
One if statement in assetResolver.ts: if (product.existingAsset && await storage.exists(key)). Everything downstream just reads the buffer. The decision point is intentionally early and intentionally simple — one place to look if reuse misbehaves.

Multi-agent orchestration likely if they explore

Why four parallel LLM reviewers instead of one mega-prompt?
Persona commitment. When you stack five personas in one system prompt, the voices average out — the legal concern gets diluted by the creative concern. Running each agent as a separate call with a focused system prompt forces each to commit to its perspective. Side benefit: parallel execution cuts wall time ~4x. Synthesis phase is a separate call that explains why each reviewer's change was accepted — that becomes the Stakeholder Review Notes panel in the UI.
What happens if a reviewer agent fails?
The rest of the orchestration proceeds. The failed reviewer is logged and excluded from synthesis input. Partial review > no review. And if synthesis fails, we fall back to the Campaign Manager's draft — which is already a valid brief.

Frontend & state likely

Why a custom hook instead of Redux / Zustand / Jotai?
ADR-004. This is a single-flow app — one pipeline, one state machine. Pulling in a state library means adding a dependency, a learning-curve tax for readers, and a file-count tax. useReducer with a discriminated-union state hits the same correctness bar without any of those costs. If this grew into multiple independent flows I'd revisit — but today, the simple thing is the right thing.
Why Zod if you already have TypeScript?
TypeScript only runs at compile time. Runtime validation at system boundaries (HTTP input, LLM output, brief parse) is the only way to catch malformed payloads before they blow up three layers deep with an opaque error. The schema IS the types — type CampaignBrief = z.infer<typeof campaignBriefSchema> — so they never drift. ADR-005.

Testing likely

How do you test LLM-powered code when the output is non-deterministic?
ADR-008. Don't assert exact strings — assert shape and invariants. For the prompt builder: "the output contains these sections, these brief fields appear verbatim, prohibited terms are in the negative list." For the pipeline: mock the OpenAI client, inject deterministic fixtures, assert the manifest shape. The RequestContext DI pattern is what makes that mock trivial.
What's your test coverage philosophy?
Inverted pyramid by importance. promptBuilder is the highest- scrutiny component, so it gets the deepest tests. pipeline is next — one end-to-end happy path + partial-failure case. Frontend components get light coverage because TypeScript + simple render assertions catch most bugs before a test would. I skip coverage percentage as a metric on purpose — it rewards the wrong behaviors on a 48-hour spike.

Infra / Docker / deployment likely if they ask you to demo live

Any war stories from building this?
Yes — I have a great one. Docker + fonts. I containerized the app and the text overlay silently broke. Gradient band drew fine, pipeline reported success, but the campaign message was invisible. Root cause: node:bookworm-slim ships zero fonts, so when @napi-rs/canvas (Skia) called fillText with sans-serif, fontconfig fell back to a zero-width glyph table. Fix: install fonts-dejavu-core + fontconfig and run fc-cache -f at image-build time. The bug was educational because the pipeline reported zero errors — the only way to see it was to look at the output PNG. Lesson: test the artifact, not just the status code.
How would this scale to 1000s of briefs per day?
Three changes. Queue: POST /api/generate returns a job ID immediately; a worker processes from a queue (SQS + Lambda, or a Vercel Cron + Upstash Redis). DALL-E tier: move off Tier 1 — the 5→3 concurrency tune was to survive Tier 1 rate limiting. Caching: the prompt template is deterministic, so identical inputs could hit a prompt-hash cache. Bonus: for certain briefs, reuse would be a shared brand-library S3 bucket instead of per-request uploads. None of these are in this codebase — they're production architecture, not a take-home.

Trade-offs & self-awareness likely as a closer

What's the biggest limitation?
Two things. One, DALL-E 3 Tier 1 p90 latency can exceed my per-image budget under load — on a bad day a 6-image brief times out. I documented it in the README rather than hiding it. Fix is a higher tier or async processing. Two, there's no real brand-safety layer between prompt and DALL-E beyond the prohibited-terms list in the template. In production I'd add a pre-generation moderation pass and a post-generation visual classifier.
What would you do differently with another week?
Three things, in priority order. One, replace the sample-brief registry with a real brief upload + brand-library UI — upload a few reference images and a style guide, then generate. Two, a prompt evaluation harness — hold-out briefs with human-scored outputs to detect regressions as I tune templates. Three, real Firefly integration behind a provider-abstraction, because Firefly's commercial-safety story is what this product actually needs in production.
If you had to cut one thing to ship earlier, what would it be?
The multi-agent brief orchestration. It's the most fun feature, but the pipeline works without it — you just lose the "stakeholder review notes" panel. Cutting it saves ~2 hours of implementation and a 10-second latency bubble. I'd keep the reuse branch, the timing chart, and the typed errors — those are load-bearing for the assignment's stated criteria.
